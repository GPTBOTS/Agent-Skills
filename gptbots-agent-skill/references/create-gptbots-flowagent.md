# Create / optimize a GPTBots FlowAgent (.bot, botType=Flow)

> Reference for the `GPTBots Skill` workflow when the target is a **FlowAgent (orchestrated agent, `botType=Flow`)**. Orchestrate the requirements into an AgentFlow built from components and connections, producing a **plaintext `.bot`** importable into the platform (`exportType=BOT`, `botType=Flow`, config in `flowRule`).

## Workflow

### 1. Requirements discussion
Clarify channels / materials / primary intents (mutually exclusive intents ‚Üí Classifier=`Branch`, judging by content ‚Üí `Condition`, judging by variable ‚Üí If-Else=`Bool`) / data to collect (multi-turn ‚Üí conversational collection, form ‚Üí form collection, cross-turn memory ‚Üí user attributes + variable assignment) / handoff method / boundaries. When optimizing, start from the `.bot` file the user provided; if they want to optimize but provided no file, ask them to export it from the platform first.

### 2. Component design (strictly follow platform rules)
Read `./flowagent-components.md` (capabilities/branches/connection rules for 14+ components) and `./variables-reference.md`. Be sure to:
- **Connect every meaningful branch**: knowledge search `no result`, Condition `else`, and the wired `exception` outlet of LLM/Condition/ChatGather/Variable nodes (when `exceptionSwitch=true`). The Classifier's `other` **and** `exception` are both **system-preset** ‚Äî you never generate either as a category. For `other`, wire its preset `branch_other` handle to a destination with the builder's `branch_other()` ‚Äî the edge MUST be `name:"_other"` + `condition:""` (the builder emits this; `name:null` makes the platform render an editable blank category instead of the built-in Other). For exception, just set `exceptionSwitch=True` on the classifier ‚Äî **there is no `branch_exception` edge**; the builder and validator reject one (`BRANCH_EXCEPTION_EDGE`). The classifier's `messages` are `[Role, ShortMemory, Input]` (no long-term memory) and `branchRunMode` defaults to `EXTRACT` ‚Äî the builder handles both. The validator guards Other with `BRANCH_OTHER_NAME` / `BRANCH_NO_OTHER`.
- **Give every Classifier category a non-empty branch rule** (and every Condition a non-empty condition): the canvas rejects an empty rule with "Cannot be empty" and the node becomes unusable.
- **Classifier rules live in the edge's `condition` (natural-language text), not in the identity prompt** ‚Äî and the handle suffix is sequential `branch_1`/`branch_2`/`branch_other` (never a timestamp id). Putting the id in `condition` makes the UI show the id instead of the rule. Use the builder's `branch_edge(rule="‚Ä¶", name="label")`; the validator flags `BRANCH_RULE_IS_ID`.
- **Prompt messages use the `text` field, never `content`** ‚Äî a `content` key imports as a BLANK prompt (silent). Each node's `messages[]` is the standard `[Role, LongMemory, ShortMemory, Plugin, (Condition), Input]` array; pass just the prompt via `add(..., prompt="‚Ä¶")` (or `role("‚Ä¶")`) and the builder assembles the rest + backfills the `Input.upstream`. KB injection = `reads_kb=True` (‚Üí `dataEnable`+`datasetMessages`), not a message. Variable assignments are `{variableName, operation, value}` with `operation` ‚àà Cover/Clear/Append. The validator flags `MSG_CONTENT_FIELD` / `MSG_ROLE_EMPTY`.
- Exactly one `Input` and one `Output`; `Output`/`Human Service` are terminal (no downstream).
- **Human Service nodes**: write `humanConfig` at the **component level** (not only bot-entity level), or the node config renders blank ‚Äî see `./flowagent-components.md` ‚Üí Human Service.
- `{{...}}` can only reference **upstream**; after replacement-type nodes (cards / LLM output), explicitly re-inject the original input with `{{start_msg_text}}` etc.
- Use If/Else (free) where appropriate; do not misuse Condition or Classifier (both are LLM-driven and consume LLM on every run).
- **Collected fields ‚Üí next step without a Variable node.** A Variable (assignment) node can only target a User Attribute / Custom Variable that **already exists in the workspace** ‚Äî importing a `.bot` does not create them, so its assignments are silently dropped ("No variables available"). Don't add Save/Variable nodes for the "collect then act" pattern: route the ChatGather collect-complete edge straight to human handoff; collected fields + conversation context carry forward and key events capture the business type/status. Also: gather fields must use `fieldName`/`showName` in JSON (use `gather_fields()`), or the platform assigns random default names; `fieldName` must be `[a-z0-9_]` (lowercase letters, digits, underscores) since it becomes a variable key.
- **A `Condition` node's IF text goes on the `conditions_true` edge** (`name:"_true"`, `condition`=IF text), with `conditions_false` as `name:"_false"`/`condition:""` ‚Äî not in a node message, or the canvas IF box shows empty. Use `condition_edges(cond, if_text, true_dst, false_dst)`; the validator guards it (`CONDITION_IF_EMPTY` / `CONDITION_EDGE_NAME`).
- **`ChatGather` is LLM-driven and monopolizes the conversation while collecting.** Its prompt's field definitions + SOP alone drive both the asking and the extraction ‚Äî no knowledge base involved. Critically, once collection starts the node **exclusively owns the dialogue** (platform behavior): mid-collection user questions ("ÊÄé‰πàÊãøÂõûÊâß/ÊÄé‰πàÊà™Âõæ" and other how-to asks) can NOT be routed to knowledge search or any other component. So anticipate the help users will need to produce each field and **embed those short how-to guides directly in the ChatGather prompt**; never design a flow that assumes a Dataset/LLM fallback can answer during collection.
- **Prompts decide runtime quality**: the Classifier (`Branch`) and `Condition` are LLM-driven, just like `LLM` components ‚Äî each `Branch` category rule and each `Condition` condition is a prompt the LLM executes on every run to route the message, so routing accuracy depends directly on how the rules are written. Make category rules mutually exclusive and unambiguous, with concrete examples of what belongs in each category when intents are easy to confuse.
- **Route with conversation context, not just the last message.** Follow-up messages are often fragments ‚Äî emotion-only replies (an emoji, a sigh), short acknowledgments, "ËøòÊòØ‰∏çË°å" ‚Äî that are meaningless in isolation but perfectly clear given the issue under discussion. Enable short-term memory on the Classifier and state in the branch rules that fragmentary/emotional follow-ups to an unresolved issue belong to **that issue's business branch**, not the fallback/"ÊóÝÊÑè‰πâ" branch; reserve the fallback for messages that stay off-topic even with context. A classifier that judges each message in isolation misroutes exactly the users who most need help ‚Äî the ones still stuck after the first answer.
- **Key events (ÂÖ≥ÈîÆ‰∫ã‰ª∂) give LLM-driven nodes long-term, cross-session business memory.** The platform auto-extracts business-valuable events from conversations into per-user structured records ‚Äî customer-defined event type, summary, entities, status (`PENDING`/`IN_PROGRESS`/`RESOLVED`/`CLOSED`), priority, confidence ‚Äî and injects recent events back into prompts. Classifier (`Branch`), `LLM`, and `Condition` components all support enabling key events in their memory settings (per-node switch + event-type selection + recent-event count, default 5; events are fetched once per session and shared across nodes), and If/Else can branch on key-event variables. Injection is implicit (auto-appended `## Recent Key Events`) or explicit via `{{key_event_<eventType>}}` placeholders. Design guidance:
  - Use key events where they earn their keep: when the user **switches topic and later returns**, or **follows up on a historical issue** from an earlier session ‚Äî the event's type/status/progress lets routing and answers pick up where things left off instead of starting cold.
  - Enable only on nodes that genuinely need event context (answering, escalation/judgment nodes); blanket-enabling inflates every node's prompt with events it doesn't use.
  - The event-type dictionary is customer-defined (name + description, ~5‚Äì10 types) ‚Äî the descriptions drive extraction accuracy, and renaming a type after launch orphans existing `{{key_event_<ÊóßÂêç>}}` references, so name types carefully up front.
  - Extraction is asynchronous (message-threshold / idle-timeout triggered), so treat events as auxiliary memory ‚Äî never the source of truth for strict-consistency decisions.
  - Docs: https://www.gptbots.ai/zh_CN/docs/tutorial/bot/user-list/key-events Write them clear, concise, and precisely executable; scope each to its node's single job; and check the whole set (including the top-level identity prompt) for conflicting goals/tone/boundaries before delivery (see *Prompt quality for LLM-capable nodes* in SKILL.md).

### 3. Reuse the existing public API to fetch real references
Same as create-gptbots-agent: use `Authorization: Bearer <API_KEY>` against the regional base URL `https://api-${endpoint}.gptbots.ai` (`sg` default, `jp`, `th`) to call `GET /v1/bot/knowledge/base/page`, `/v1/database/tables/page`, etc., to fetch real ids and write them into Dataset/ToolApi/Workflow components; if not found, leave them blank. **Never call internal/console APIs**.

### 4. Generate
- **Use the builder script instead of hand-writing JSON**: write a generation script that imports `FlowAgentBuilder` (plus `role/user_input/kb_msg/cond_msg/mem/ke/gather_fields/var_cfgs` helpers) from `../scripts/build_gptbots_flowagent.py` ‚Äî it auto-generates component ids, the strict edge handles (and rejects key-repeating suffixes that distort canvas lines), branch ids, and layout, and `save()` runs the validator. Factor repeated node shapes into helper functions; regenerate from the script on every revision. (`python3 ../scripts/build_gptbots_flowagent.py` prints usage; `--demo <dir>` emits a validated example.)
- **Keep the prompts external, one md file per node** (the standard for every bot): a `prompts/` folder with `<node name>.md` per component (identity, each classifier/condition rule, each answer LLM, each gather SOP). Load with `load_prompts("prompts/")` (re-exported by the builder, from `../scripts/gptbots_prompts.py`) and reference by name ‚Äî `messages=[role(P["Intent_Classifier"]), user_input()]`. Name each file after the component's `name` so a renamed/missing prompt fails loudly via `load_prompt_store("prompts/").require(name)`. This keeps every prompt reviewable and editable on its own, free of Python/JSON escaping, and separate from the flow structure. (A single `prompts.md` of `## key` sections is still accepted for compatibility, but use the folder for new bots.)
- Top level: `formatVersion`, `exportType=BOT`, `exportTime` (epoch-ms integer Long, NOT an ISO string ‚Äî strings fail import), `name`, `botType=Flow`, `flowRule.components[]`.
- **Each component's `type` MUST be the `FlowComponentType` enum value, not the UI name** (e.g. Classifier‚Üí`Branch`, If-Else‚Üí`Bool`, Knowledge Search‚Üí`Dataset`, Card Message‚Üí`Predefine`, Tools‚Üí`ToolApi`, Human Service‚Üí`Human`). See the mapping table in `./flowagent-components.md`. A UI name like `"Classifier"` will fail import.
- Component `id` must be unique; `nextComponents[].nextComponentId` must point to an existing component.
- **Strong-typed Integer fields are import-fatal if mistyped** ‚Äî the backend strictly parses `components[].id`, `x`, `y`, `nextComponents[].id`, `nextComponentId`, `sort` as Integer; ANY string (`"e1"`, `"vueflow__edge-..."`, even a quoted `"1"`) fails with `value X is not allowed for field "‚Ä¶"` (same parsing as `exportTime`). Extra/unknown fields are tolerated (`FAIL_ON_UNKNOWN_PROPERTIES=false`) ‚Äî only wrong types kill the import. Edge ids: unique integers not colliding with component ids (e.g. `100000+seq`); the builder generates these automatically. Before delivery, recursively scan the JSON and convert any quoted number / string in these positions to a bare integer (see the strict-type table in `./flowagent-components.md`).
- **Set each component's fields & enum values, and build connection handles correctly** ‚Äî see *Per-component JSON fields & enum values* and *Connections & handles* in `./flowagent-components.md`. Every `nextComponents[]` edge needs `sourceHandle = right{thisId}-{key}[_suffix]` and `targetHandle = left{nextComponentId}-{key}` with keys matching each component type; a wrong handle id/key still imports but renders a **distorted/misrouted line** on the canvas. Any out-of-enum value (e.g. `contentType`, `reasoningEffort`) fails import.
- Leave model id blank (backend backfills); leave plugin/tool authentication and cross-organization references blank (cleared on import).
- **Always include top-level `"multiModal": {"multiModalInput": {}}`** ‚Äî the import does no backfill and the console auto-save NPEs (HTTP 500 on every save) if `multiModalInput` is null (backend regression 2025-12-02); the builder emits this automatically. Don't guess the enum fields inside; copy from a real export or omit them.
- **Always (re)generate a mermaid flow diagram of the design** and write it into a `## Flow (mermaid)` section of an `overview.md` delivered next to the output file, so the design intent is reviewable.

### 5. Quality check (mandatory)
```
python3 ../scripts/validate_gptbots_config.py <name>.bot
```
Non-zero exit code ‚Üí fix per `errors` ‚Üí rerun, until it passes before delivery.

### 6. Delivery
Place the new/updated `.bot` file and its `overview.md` (including the mermaid diagram) in the current working directory and return their local paths (never overwrite the user's original file unless asked ‚Äî deliver an updated copy alongside it). Then tell the user: on **www.gptbots.ai** (developer space), **Create Agent / Workflow ‚Üí Import**, then select the file.
